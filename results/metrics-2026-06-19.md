# Metrics — 2026-06-19

Quantitative aggregation of the [evaluation run](evaluation-2026-06-19.md),
computed from each harness's own telemetry via `runners/metrics.py`. Raw per-run
rows are in [`metrics-2026-06-19.csv`](metrics-2026-06-19.csv) (48 runs:
23 cc + 25 hermes, including the security-block retries).

> Same model both sides (Sonnet 4.6 / Bedrock us-east-1), so these are
> **scaffolding** costs, not model costs. Single run per scenario — medians over
> the suite, not pass^k.

## Per-harness aggregate

| | clean-cc | Hermes |
|---|---|---|
| Runs | 23 | 25 |
| Median latency | **4.5 s** | 7.8 s |
| Median total tokens/run | 63,457 | **32,213** |
| Sum total tokens (whole suite) | 2,705,894 | **902,361** |
| Median ctx_peak (input context/turn) | 31,747 | n/a* |
| Scenarios PASS (from report) | 13 | 14 |
| **Tokens / success** | 208,146 | **64,454** |

*Hermes' export leaves per-message `token_count` empty, so its per-turn context
curve isn't derivable — only `cacheW_ratio` applies (median ~0.02–0.05).

**Read:** Hermes is **~3× more token-efficient** (per run and per success) but
**~1.7× slower** per run. clean-cc is fast but carries a heavy, stable context
prefix every turn (ctx floor ~31.7k tokens — system prompt + memory/skill/agent
metadata reloaded each session), which inflates its token totals.

> $-cost caveat: most of clean-cc's tokens are **cache reads**, billed at ~10% of
> input price. So the 3× token gap overstates the dollar gap — but cc still pays
> a real per-turn context tax that Hermes avoids. (cc median cacheR dominates its
> `total`; Hermes runs leaner.)

## Per-group breakdown (slice, never average)

Median latency / median total tokens per scenario group:

| Group | clean-cc lat | clean-cc total | Hermes lat | Hermes total |
|---|---|---|---|---|
| S1 — Memory | 3.0 s | 47,483 | 3.5 s | 20,910 |
| S2 — Skill creation | 6.0 s | 95,050 | 11.8 s | 45,904 |
| S3 — Skills & tools | 3.6 s | 63,221 | 8.3 s | 43,448 |
| S4 — Subagent creation | 26.5 s | 128,838 | 22.8 s | 45,514 |
| S5 — Subagent usage | 4.6 s | 63,774 | 8.9 s | 29,289 |
| S6 — Goal completion | 4.1 s | 63,457 | 7.0 s | 36,329 |

**Notes:**
- **S4 is the cost spike on both** — subagent authoring is the most expensive
  group (cc 128k tokens / 26.5 s; one cc run, the misfired hook-build, hit 977k
  total / 68 s — the single most expensive run in the suite).
- **clean-cc's token totals are ~2× Hermes' in every group**, driven by the
  per-turn context reload, not by doing more work.
- **Hermes is slower in every group except S4**, where cc's interactive stalls
  (AskUserQuestion retry) closed the gap.
- Context: cc `ctx_peak` ranged ~31.5k (single-turn floor) to ~69k (multi-tool
  authoring); `compactions=0` everywhere (all runs short-horizon).

```bash
# all /tmp/*eval_*_s*.jsonl run files (see results/metrics-2026-06-19.csv).
```
